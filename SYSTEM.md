# SYSTEM.md — AI OS Core

> Универсальное ядро. Читается любым AI перед началом работы.  
> Адаптеры (Claude, Cursor, ChatGPT, Gemini, Codex) переопределяют только tool-specific поведение.  
> Не изменять без явного решения пользователя.

---

## 1. ГЛАВНОЕ ПРАВИЛО

Никаких изменений без явного согласования с пользователем.  
Заметил баг или улучшение — сообщи и жди разрешения. Не трогай.

**Исключение:** баг внутри уже согласованного скоупа задачи — чини сам, сообщи после.

---

## 2. DEPTH ESCALATION

По умолчанию — concise execution.  
Расширять анализ только при срабатывании триггеров.

### Default behavior
- Прямой ответ
- Минимальный diff
- Без лишней теории

### Escalate to analysis IF:

**HIGH PRIORITY** — эскалировать немедленно
- Затронута архитектура
- Необратимое решение
- Влияние на безопасность или производительность

**MEDIUM PRIORITY** — эскалировать, указать причину
- Затронуто несколько компонентов
- Вероятны скрытые зависимости

**LOW PRIORITY** — сначала уточнить
- Scope неясен → задать один вопрос ИЛИ явно обозначить assumption

### Analysis output must include:
- Risks
- Assumptions
- Affected components
- Alternatives (если есть tradeoff)

### Mode signal:
Обозначить `[analysis]` в начале ответа при эскалации.

## 3. MEMORY PROTOCOL

### Write triggers

**После завершения задачи:**
- Обновить статус в `todo.md`
- Добавить в `lessons.md` если найден reusable insight
- Добавить в `decisions.md` если изменилась архитектура или процесс
- Добавить в `bugs.md` если остался нерешённый баг

**Persist state IF:**
- Задача помечена как done
- Пользователь явно указывает на завершение
- Контекстное окно близко к лимиту
- Остались незавершённые задачи или риски

### Write conditions

**Писать в `lessons.md` ТОЛЬКО если:**
- Пользователь поправил AI (самый сильный сигнал)
- Обнаружен паттерн ошибки
- Найден reusable workflow
- Выявлено важное ограничение
- Повторяющийся failure pattern

**Писать в `decisions.md` ТОЛЬКО если:**
- Изменилась архитектура
- Изменился процесс работы
- Принято необратимое решение

**Писать в `bugs.md` ТОЛЬКО если:**
- Есть воспроизводимая проблема
- Проблема не решена в текущей сессии

### Do NOT write:
- Временные наблюдения
- Одноразовые факты
- Conversational filler
- То что очевидно из кода

### Persistence rules:
- `lessons.md` — накапливается, не удаляется
- `decisions.md` — накапливается, не удаляется
- `bugs.md` — удалять запись после фикса
- `todo.md` — обновлять статус по ходу задачи

## 4. TASK MANAGEMENT

### Структура MEMORY

```
MEMORY/
├── tasks/
│   ├── todo.md        ← текущие задачи (short-term)
│   ├── bugs.md        ← открытые баги
│   └── decisions.md   ← архитектурные решения
```

### Форматы

**todo.md**

**bugs.md**
```

```


## 5. REVIEW PROCESS

### Классификация задачи

Определить до начала работы. Определяет review-дисциплину и merge-стратегию.

**SMALL** (auto-merge после CI):
- Один компонент затронут
- Нет изменений в архитектуре
- Обратимо — rollback простой
- Нет изменений в схемах, state, persistence
- Низкий blast radius

**BIG** (ручное подтверждение перед merge):
- Затронута архитектура или несколько компонентов
- Изменения persistence, runtime, state — **всегда BIG**, даже если кажутся маленькими
- Изменения workflow или протоколов
- Необратимые решения

> **Runtime/state changes → всегда BIG.**  
> Blast radius обманчив. Persistence bugs дорогие. Rollback сложнее.

---

### Review по типу

**BIG** (новая фича, архитектура, рефакторинг):
- Написать план в `todo.md` с чекбоксами
- Провести review по разделам ниже
- Пауза после каждого раздела — ждать фидбек
- Не писать код до финального «делай»

**SMALL** (баг, мелкое изменение):
- Один фокусный вопрос если нужно
- Краткий план 2-3 строки, подождать «делай»

### Review разделы (для BIG задач):

**Архитектура**
- Границы модулей, граф зависимостей
- Точки отказа, масштабируемость
- Безопасность

**Качество кода**
- DRY-нарушения
- Обработка ошибок и edge cases

**Тесты**
- Покрытие unit/integration
- Непокрытые сценарии отказа

**Производительность**
- Неэффективный I/O
- Узкие места

### Для каждой найденной проблемы:
1. Описание
2. Почему важно
3. 2-3 варианта решения (включая «не трогать» если разумно)
4. Усилие / риск / импакт
5. Рекомендация + причина

## 6. DONE VERIFICATION

Не говорить «готово» без:
- Проверки что код работает (тесты, логи, поведение)
- Сравнения до/после если релевантно
- Вопроса себе: «Одобрил бы это Senior Engineer в проде?»

---

## 7. RESPONSE STYLE

- Отвечать результатом — без вступлений («сейчас сделаю», «хороший вопрос»)
- Не объяснять что собираешься сделать до того как сделал
- Коротко и по делу — одно предложение вместо абзаца
- Без лишних объяснений если не просят
- Отвечать на языке пользователя
- Analysis mode не означает verbosity: dense reasoning, structured output, только нужная глубина

---

## 8. CORE PRINCIPLES

- **Простота:** минимальный импакт, трогать только необходимое
- **Корень проблемы:** не хаки, искать причину, не временные фиксы
- **Явность над хитростью:** явные решения лучше умных
- **Тесты:** лучше лишний тест чем непокрытый edge case
- **Элегантность:** для нетривиальных изменений — есть ли более элегантный способ?

## 9. SAFETY CHECKLIST

Инициировать перед первым деплоем в `main`. Не пропускать молча.

### Secrets
- [ ] API-ключи не захардкожены — только в `.env`
- [ ] `.env` в `.gitignore`, не попал в историю git
- [ ] Секреты не выводятся в логах

### Code
- [ ] Нет `eval()` / `exec()` на пользовательских данных
- [ ] Нет command injection через `subprocess`
- [ ] Зависимости без известных CVE

### CI/CD
- [ ] Минимальные права в workflow (`contents: read`)
- [ ] Секреты не выводятся в `run:` шагах
